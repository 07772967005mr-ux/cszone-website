2.3.2 — Testing
Component 2: Programming  ·  ~22 min

Why Test?

Testing verifies that a program works as expected and produces correct outputs for all inputs — including valid, invalid, and edge case inputs. Testing identifies logic errors and unexpected behaviour before a program is deployed.

Key distinction: Errors found during testing are fixed. Errors found by end users after deployment are expensive and damage trust.

Types of Test Data

For any input, three categories of test data should be used:

TypeDescriptionExample (age, valid 0–100)
NormalTypical, valid data within expected range25, 40, 67
BoundaryValues at the exact limits of acceptable range0, 1, 99, 100
ErroneousInvalid data outside range or wrong type — program must handle/reject it-1, 101, "hello"

Why boundary data matters:

Most errors occur at the limits — off-by-one errors are very common. Testing 0 and 100 (the exact boundaries) checks the condition operators (<, <=, >, >=) are correct. Also test 1 and 99 (just inside) and -1 and 101 (just outside = erroneous).

Types of Testing

Iterative testing

Testing is done throughout the development process — each small section is tested as it is written. Errors are found and fixed early, when they are cheapest to fix. Also called "testing as you go".

Final (terminal) testing

The complete program is tested as a whole at the end of development — checking all components work together correctly. Also called system testing or acceptance testing.

Alpha testing

Carried out by the development team (internal testers) before release to the public. Identifies bugs in a controlled environment.

Beta testing

A limited release to a selected group of real users outside the development team. Identifies bugs under real-world usage conditions that the developers didn't think to test.

Trace Tables

A trace table is used to manually track the values of variables at each step of an algorithm. It is used to identify logic errors by showing exactly what happens at each iteration or step.

Example: trace table for a loop summing 1 to 4

Common Mistakes

  • Confusing boundary with erroneous — boundary is valid (at the limit); erroneous is INVALID (outside the limit)
  • Only testing normal data — exams specifically ask about boundary and erroneous test data
  • Confusing alpha (internal/dev team) with beta (external/real users)
  • Saying iterative testing is done at the end — it is done THROUGHOUT development
  • Thinking testing proves a program is correct — testing can only find errors, not guarantee correctness
